## 一、项目背景

用户是一名算法工程师，独立发起了一个 AI 眼镜助手项目（代号 Jarvis）。核心产品形态：

- **硬件**：AI 眼镜，全天候录制第一人称视角视频
- **软件**：AI 助手，通过语音/文字交互，能主动回忆、查互联网、查天气
- **训练路线**：SFT + RLHF，基座模型 7B 级别
- **定位**：偏向生活场景（非工作助手），自然对话，没有"AI感"，像朋友一样

### 真实系统架构

1. **边缘截流**：硬件每 3 秒抓一帧，本地 NPU 做 SSIM/光流比对，相似度 >95% 的丢掉（比如盯着屏幕不动）。每天约 20000 帧截流到约 1000 帧关键动作帧
2. **VLM 图生文**：关键帧送云端 VLM，转为文字描述
3. **语义合并**：如果当前帧的文字描述和上一条语义相似，只更新 end_time；否则新建一条。防止向量数据库被重复描述撑爆
4. **三层记忆**：
   - **原子层**（像素级）：截流+VLM+合并后的记录，用于极端细节查询（"钥匙放哪个口袋了"）
   - **事件层**（情节级）：**核心！** 每 2-3 小时，把原子层发给便宜 LLM（如 Qwen-Turbo）总结成事件块，包含因果逻辑。RAG 查询的主力命中目标。单独的向量库 collection
   - **每日总结**（主题级）：每天从事件层提炼情绪、偏好、未完成目标。**唯一进入 System Prompt 的数据**，7B 模型始终可见

### 三层记忆的关键区别

不只是粒度不同，而是**生成方式、存储位置、查询目的**都不同：

| 层级 | 生成方式 | 存储位置 | 查询目的 |
|------|---------|---------|---------|
| 原子层 | 边缘截流→VLM→语义合并 | 向量库（独立collection） | 极端细节查询 |
| 事件层 | 每2-3h 便宜LLM总结原子层 | 向量库（独立collection） | **RAG主力** |
| 每日总结 | 每天LLM从事件层提炼 | **System Prompt** | 始终可见，全局状态 |

## 二、所有关键决策及讨论过程

### 2.1 大厂工程师的工作流程（用户提问）

用户问："如果是大厂工程师负责这个项目，工作顺序是什么？"

最终确定的流程：
1. **Phase 0**：定义能力边界和评估标准
2. **Phase 1**：写 Golden Examples（高质量范例）
3. **Phase 2**：设计并生成 SFT 数据
4. **Phase 3**：SFT 训练
5. **Phase 4**：RLHF

用户当时已经跳了步骤，直接在用 LLM 合成对话数据。

### 2.2 能力边界定义

AI 助手只有文字输出，不做真实世界操作（不订机票、不控制设备）。6 个核心能力：
1. 记忆回忆（query_memory）
2. 记住信息
3. 互联网搜索（query_internet）
4. 天气查询（query_weather）
5. 主动提醒（基于状态表）
6. 情感陪伴

### 2.3 System Prompt 设计

**最重要的决策之一。** 经过多轮讨论和纠正：

- Claude 初始方案：长 system prompt，包含角色定义、能力边界、工具定义、行为规则等
- 用户反驳：**"为什么把工具定义放在 system prompt 里？这些可以通过训练数据内化。"**
- 最终结论：**System Prompt = 用户状态表（3 个模块）**，其他全部通过 SFT 数据内化
- 理由：3 个固定工具，参数少，7B 模型完全可以通过训练学会。只有动态的用户状态表必须在运行时注入

**System Prompt 三模块设计**（用户提出）：
1. **Block 1 - 用户画像**：长期静态信息（姓名、职业、饮食偏好、健康状况等）
2. **Block 2 - 实时遥测**：秒级动态信息，客户端上报（当前位置、动作、环境）
3. **Block 3 - 主动钩子/洞察**：中期动态，后端大模型异步提取（情绪变化、偏好漂移、未完成目标）

Block 3 是关键创新——将原始数据转化为"洞察"，让 7B 模型直接拿到预消化的结论，不需要自己做模式识别。

**另一个关键纠正**：Claude 假设产品是单用户的。用户纠正："这个产品会卖给不同用户，每个用户都要好体验。"因此模型必须学会阅读和适配任何状态表，而非记住特定用户。

### 2.4 状态表字段取舍

- Claude 建议：去掉低频字段省 token
- 用户反驳：**去掉信息就意味着模型不能给出"哇"的个性化响应**（比如推荐美食需要知道饮食偏好、健康状况、家乡口味、最近减肥目标）
- 最终决定：**保留所有字段，只优化格式**（把嵌套 JSON 拍平省 token）
- "哇时刻"来自训练数据质量，不是架构技巧

### 2.5 SFT 数据管道问题

用户发现现有数据管道有严重问题：

1. **维度强制**：随机分配维度组合然后让 LLM 生成，导致不自然、矛盾的数据
2. **没有真实 RAG 数据**：`query_memory` 的返回是 LLM 编造的，不是来自真实的记忆系统
3. **训练-推理不一致**：训练数据的工具输出是 LLM 编造的完美结果，但真实部署时记忆数据来自视频转文字，会有噪声和不完整

**解决方案**：先建模拟记忆数据，再用它生成训练数据。工具输出必须来自真实模拟数据。

### 2.6 记忆数据格式

最终决定用**自然语言**（不是 JSON），三层不同粒度：
- 原子层：`时间: ... 位置: ... 动作: ...`
- 事件层：`时间: ... 事件: ...`
- 每日总结：`日期: ... 总结: ...`

### 2.7 用户数量和反过拟合策略

- 初始：6 个用户 → 用户担心过拟合
- 修订：12 个 → 仍担心
- **最终：30 个用户**，每人随机 14-30 天
- 理由：SFT 教的是新行为（工具调用、记忆策略），泛化靠基座模型；但用户自适应语气需要多样性
- 反过拟合策略：更多样的用户 + 更短的历史 > 更少的用户 + 更长的历史

### 2.8 30 个用户画像分布

| 群体 | 数量 |
|------|------|
| 年轻职场人 25-35 | 8 人 |
| 大学生 18-24 | 4 人 |
| 中年职场/家长 35-50 | 5 人 |
| 自由职业/创作者 25-40 | 3 人 |
| 银发族 60+ | 3 人 |
| 运动/户外爱好者 25-45 | 3 人 |

性别 15 男 / 15 女，城市从一线到县城，健康状况覆盖痛风、过敏、高血压、糖尿病、焦虑症、孕期、术后恢复等。

### 2.9 生活剧本生成

- P01（陈逸飞，程序员）的手写剧本作为种子
- 其余 29 人用 LLM 生成，**不给 few-shot 示例**（用户纠正："你生成第一个的时候也没有示例参考，我要完全一样的方法"）
- 剧本包含：核心设定、居住环境（有画面感）、工作日/休息日作息、社交模式、近期事件线（12-15条，有关联性和连续性）、说话风格、Jarvis 高频场景

### 2.10 记忆数据生成 — 随机事件设计

- 初始方案：固定事件池 + 权重随机
- 用户指出问题：固定池可能跟用户画像矛盾（比如没养猫的人出现猫事件）
- **最终方案**：完全去掉固定事件池，让 LLM 根据每个用户的画像生成个性化随机事件

### 2.11 生活剧本的代码问题

- `life_scripts.py` 最初用一个巨大字典（146KB/2435行），导致 PyCharm 解析器溢出
- 部分剧本有语法错误：字典语法 `"今天": "xxx"` 错误地用在列表里
- **解决方案**：拆成独立赋值语句 `life_scripts["P01"] = {...}`、`life_scripts["P02"] = {...}`

## 三、三步流水线重构（最新改动）

### 背景

原始 `generate_memory.py` 一次性让 LLM 生成三层记忆，跟真实系统架构不符。

### 新架构：三步流水线

| 步骤 | 说明 | 每天LLM调用 |
|------|------|-----------|
| Step 0 | 生成事件日历（保留原有逻辑） | 1次/用户 |
| Step 1 | 原子层：事件驱动格式 `{start, end, location, action}`，模拟语义合并 | 1次/天 |
| Step 2 | 事件层：原子层按2-3小时分块，每块LLM总结 | 4-8次/天 |
| Step 3 | 每日总结：从事件层提炼情绪/偏好/未完成目标 | 1次/天 |

### 原子层的关键约束（经用户纠正）

用户看了第一版原子层后指出问题：**和真实视频转文字结果不匹配**。

**问题**：
1. 一条记录里做了多件事（"起床给猫加猫粮换水，蹲厕所刷手机"）——真实 VLM 每条对应一个画面变化，不会合并不同活动
2. 包含非视觉信息（"听到闹钟响"、"听到猫叫"）——摄像头看不到声音
3. 描述更像事件层而非原子层

**修正后的规则**：
- **纯视觉**：只能描述摄像头能看到的内容。禁止听觉、内心活动、屏幕内容
- **一条一个动作**：不同活动必须分开记录
- **语义合并**：只合并完全相同的连续动作（如坐工位看屏幕 2 小时合为一条）
- **action 控制在 15 字以内**
- **一天约 100-200 条**

**修正前后对比**：

| 指标 | 修正前 | 修正后 |
|------|--------|--------|
| 条数 | ~45 条 | 136 条 |
| 每条内容 | "起床给猫加猫粮换水，蹲厕所刷手机" | "往猫碗里倒猫粮" |
| 信息类型 | 混入听觉/内心 | 纯视觉："伸手拿手机" |
| 工作时段 | "写代码和Review代码和参加站会" | "面对电脑屏幕，双手敲键盘" |

## 七、用户的沟通风格和偏好

- **先小规模测试再全量**（省 token）：比如先跑 1 天看效果，确认质量后再跑全部
- **喜欢被引导而非被动执行**："你能不能像大厂资深工程师一样引导我？"
- **会直接指出问题**："为什么你想不到这些？""这个和真实系统不匹配"
- **自学者心态**："我不是大厂工程师，我是自学的，这个项目是独立发起的"
- **注重真实性和一致性**：反复强调训练数据要和真实部署场景一致
